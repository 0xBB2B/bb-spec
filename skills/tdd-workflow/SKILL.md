# 测试驱动开发（TDD）纪律

适用于：**所有语言、所有业务逻辑代码**的新增、修改、删除变更。

> 核心理念：**测试是设计工具，不是验证工具。** 先写测试，迫使你以"调用方视角"想清楚行为契约——这是 TDD 区别于"补测试"的本质。事后补的测试只能验证"代码做了什么"，无法暴露"代码该做什么"。

## 0. 触发场景

**TRIGGER**（命中任一即应应用本纪律）：

- 新增、修改、删除任何业务逻辑代码源文件：
  - **后端**：`.go` / `.py` / `.java` / `.kt` / `.rs` / `.rb` / `.cs` 等
  - **前端**：`.ts` / `.tsx` / `.vue` / `.js` / `.jsx` / `.svelte` 等
  - **脚本**：可执行 `.sh` / `.bash` / shell function 等（含业务逻辑的）
- 用户口语触发：
  - "TDD" / "test-driven" / "测试驱动"
  - "写测试" / "加测试" / "补测试"
  - "test-first" / "先写测试"

**SKIP**（以下情况本纪律不适用，无需先写测试）：

- **纯定义类文件**：
  - 仅含 struct / interface / enum / type alias 定义
  - 仅含常量 / 全局变量 / 枚举值定义
  - 仅含 import 重新导出
- **生成代码**：`*_gen.go` / `*.pb.go` / protobuf / OpenAPI / GraphQL codegen 产物等
- **纯文档**：README / 注释 / 设计文档
- **纯配置**：`*.yaml` / `*.toml` / `*.json` 等配置文件（无嵌入业务逻辑）
- **第三方代码**：`vendor/` / `node_modules/` 等

## 1. Red-Green-Refactor 循环（强制）

**所有代码变更**必须严格按以下三阶段执行，**不得跳过**任何一步：

```
┌─────────────────────────────────────────────────────────────┐
│  RED      → 先写一个失败的测试，明确行为预期                  │
│             跑测试 → 确认 FAIL（且失败原因是符合预期的）       │
├─────────────────────────────────────────────────────────────┤
│  GREEN    → 写最小实现让测试通过                              │
│             跑测试 → 确认 PASS（且没有破坏其他用例）           │
├─────────────────────────────────────────────────────────────┤
│  REFACTOR → 在测试保护下重构（去重、命名、抽象）              │
│             跑测试 → 确认仍然 PASS（每次小改动后都跑）        │
└─────────────────────────────────────────────────────────────┘
              ↓ 进入下一个最小行为单元
```

### 1.1 RED 阶段的要点

- 测试必须**真的失败**（不能因为编译错误失败，而是因为断言不通过）
- 一次只引入**一个**最小行为预期；多个预期分多轮 RED-GREEN
- 失败信息要清晰指向"哪个行为没被满足"，不是"undefined symbol"

### 1.2 GREEN 阶段的要点

- 写**最小**实现让测试通过——允许"傻瓜实现"（如 `return 5;` 让一个固定输入用例通过）
- 暂时**不要**为未覆盖的输入提前写代码，等下一轮 RED 推动
- 不要在 GREEN 阶段做重构，那是 REFACTOR 阶段的事

### 1.3 REFACTOR 阶段的要点

- 在测试**全绿**前提下进行：去重、命名优化、抽小函数、调整结构
- 每次小改动后立刻跑测试
- **不要**在 REFACTOR 阶段添加新功能（那需要回到 RED）

## 2. 三种变更场景的标准流程

### 2.1 新增功能

```
1. RED   写失败测试，描述新功能的最小行为
2. RED   跑测试 → 确认 FAIL
3. GREEN 写最小实现
4. GREEN 跑测试 → 确认 PASS
5. REFACTOR 重构（如需要）
6. 回到第 1 步，引入下一个行为预期
```

**禁止**：先写实现再补测试。

### 2.2 修改功能

```
1. RED   先修改 / 新增测试用例，反映新行为预期
2. RED   跑测试 → 确认 FAIL（因为实现还是旧行为）
3. GREEN 修改实现让新测试通过
4. GREEN 跑测试 → 确认所有测试 PASS（含未改的旧测试）
5. REFACTOR 重构（如需要）
```

**禁止**：直接改实现，再被动调整测试让它通过——这是"测试反向适配实现"，丢失了测试的设计价值。

### 2.3 删除代码

```
1. 检查 删除前先确认无其他模块依赖（grep 引用 / 编译检查 / 类型检查）
2. RED   先删除对应的测试用例
3. GREEN 再删除实现代码
4. 跑完整测试套件 → 确认无连带破坏
```

**禁止**：先删实现导致测试编译失败，再被动删测试——这种顺序会掩盖"实际上还有别处依赖"的事实。

## 3. 测试覆盖率期望

### 3.1 量化目标

- **整体期望**：测试覆盖率维持在 **80% 以上**
- 这是**期望**而非**硬性拦截**：变更显著拉低覆盖率时，应在 commit / PR 说明中解释原因或补齐测试

### 3.2 覆盖优先级

并非所有代码都需要 80%+ 覆盖。按以下优先级保证覆盖：

1. **业务逻辑层（最高）**：service 层 / domain 层 / use-case 层
2. **关键纯函数**：算法、计算、转换、校验类
3. **错误分支**：所有 `if err != nil` / `try/catch` / `match err` 分支
4. **边界条件**：空输入、零值、最大值、并发竞态
5. **集成边界（最低）**：repository / external API client 用集成测试覆盖更划算

### 3.3 不强求覆盖的部分

- 纯定义类文件（参见 §0 SKIP 列表）
- 框架样板代码（如 main 函数、依赖装配代码）
- 一次性数据迁移脚本

### 3.4 覆盖建议

- 代码变更**应尽可能附带**对应测试
- 若确有困难（重构、修 bug 难以新增用例等），需在 commit / PR 说明中**注明原因**——但**不作为硬性拦截**条件
- 若变更**反而提高**了覆盖率（如重构去除死代码），值得在 PR 描述中提一句

## 4. 测试优先于实现（行为契约）

### 4.1 需求不清晰时的处理

遇到需求不清晰时，**不要**直接猜测实现，而应：

1. 先**写测试**来明确行为预期（"调用 X 期望返回 Y"）
2. 让测试**显式表达**你理解的需求边界
3. 把测试拿给用户确认（"我理解的需求是这样的，对吗？"）
4. 用户确认后再实现

### 4.2 测试作为活文档

写好的测试应该满足：

- **可读**：测试名直接描述行为（`Test_用户名为空时_应返回参数错误` 比 `TestUserService1` 强）
- **独立**：测试之间不依赖执行顺序
- **可重现**：随机性、时间、IO 都被注入或固定
- **快**：单测应在毫秒级完成（慢测应归类为集成测试）

### 4.3 禁止的反模式

- ❌ 先写实现再补测试（测试只能复述实现，无法暴露设计问题）
- ❌ 用超大集成测试代替基础单测
- ❌ "函数被调用一次"当通过——没有真正的行为断言
- ❌ 只覆盖 happy path，不写错误分支
- ❌ 测试失败时直接放宽断言让它通过（应该回头审视实现或需求）

## 5. 与语言特化 testing skill 的关系

本 skill 是**通用纪律**（什么时候 / 按什么顺序写测试）。各语言的**惯用法、测试组织、工具链**由特化 skill 覆盖：

| 语言 | 特化 skill | 补充覆盖 |
|---|---|---|
| Go | `golang-testing` | table-driven tests、subtests、benchmarks、fuzzing、coverage 命令、`_test.go` 命名约定 |
| 其他语言 | （暂未建立） | 按语言惯例（Pytest fixtures、Jest mock、Vitest 等） |

**协作原则**：本 skill 决定"做不做、按什么顺序做"；特化 skill 决定"具体怎么写"。

## 6. 自检清单

每次代码变更前/中/后对照：

**变更前**：
- [ ] 已识别变更类型（新增 / 修改 / 删除）
- [ ] 已确认目标文件不在 SKIP 列表（§0）

**RED 阶段**：
- [ ] 已先写 / 改测试（不是先动实现）
- [ ] 已跑测试**确认 FAIL**（不是想象 FAIL）
- [ ] 失败原因符合预期（断言失败而非编译错误）

**GREEN 阶段**：
- [ ] 实现是**最小**可行（没有提前写未被测试覆盖的代码）
- [ ] 跑测试**确认 PASS**（含历史用例）

**REFACTOR 阶段**：
- [ ] 重构期间**没有**新增行为
- [ ] 每次小改动后**都跑了**测试

**变更完成**：
- [ ] 覆盖率未显著下降（或已在说明中解释）
- [ ] 删除场景下，已通过完整测试套件确认无连带破坏
