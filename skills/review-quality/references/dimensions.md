# 质量审视维度速查

本文件被主流程在 SKILL.md §3 "并行派工" 使用：

- 主对话读本文件、抽取对应维度小节、转写为可执行的搜索/检查动作后注入 agent prompt
- **agent 不读本文件**，只看主对话注入的清单（避免重复加载）
- agent 划分按 SKILL.md §3.1–§3.5（5 个 agent，多个维度可合并到同一 agent）

每个维度的格式统一：

```
## <维度名>
- 关注点：
- 检查项：
- 反例（命中即报）：
- 不报的情形：
```

**所有维度共享的硬性要求**：

- 每条 finding 必须有 `file:lines` 与原文证据
- 严重度按主 SKILL.md §3.6 判定（🔴 BLOCKER / 🟡 IMPORTANT / 🟢 NIT）
- 优先核对项目自身的"关键约束"（来自 SKILL.md §2 项目档案摘要），通用规范作为补充

---

## 1. 架构与分层一致性

- **关注点**：本仓库声明的分层是否在代码里被遵守；是否存在越权调用、错位职责
- **检查项**：
  - 项目档案摘要中的关键约束（架构 / 分层相关条目）是否被遵守
  - handler / 控制层是否写了业务规则、直接访问数据层
  - service / 业务层是否写了原始 SQL、协议层细节、框架对象（如 `gin.Context`）
  - repository / 数据层是否承载业务规则、是否返回 HTTP DTO
  - 是否引入未授权的分层名称（如 manager / facade / usecase 等，若 CLAUDE.md 禁止）
  - 跨服务/跨模块调用是否走声明的边界（如 gRPC metadata 注入而非业务字段）
  - 编排逻辑是否分散到多个领域 service（应集中到编排 service）
- **反例**：
  - handler 里 `if x < 18 { return 400 }` 的业务判断
  - service 里 `db.Query("SELECT ...")` 直接拼 SQL
  - 领域 service 持有别领域的 repository
  - 在 controller 层处理事务（`db.Begin()` / `tx.Commit()`）
- **不报的情形**：
  - 命名差异但职责正确（如把 repository 叫成 dao，但仅是命名风格）
  - 单文件示例 / 一次性脚本 / migration 工具

## 2. 测试覆盖与质量

- **关注点**：测试是否真正验证了行为；是否存在反向破坏生产设计的测试技巧
- **检查项**：
  - 项目档案摘要中的关键约束（测试相关条目）是否被遵守
  - 关键 service / 核心纯函数 / 错误分支 / 边界条件是否有用例
  - 用 `var fnName = realFn` / `var timeNow = time.Now` 等 monkey-patch 风格替换的位置
  - 仅断言 "函数被调用一次" / 没有断言的测试
  - 只覆盖 happy path，缺失错误分支
  - 集成测试是否覆盖对外暴露的 RPC / 路由（每个至少一个成功路径 + 关键失败路径）
  - mock 是否硬编码返回值（应能按用例动态配置）
  - 测试是否反向修改业务语义以让自己通过
  - 临时关闭的 / 长期 skip 的测试
- **反例**：
  - `var httpGet = http.Get` 仅为可替换
  - `assert.NotNil(result)` 后无后续断言
  - `t.Skip("TODO: fix flaky")` 长期存在
  - 集成测试启动了真实下游服务而非 mock
- **不报的情形**：
  - 单纯的 struct / interface / 常量定义文件没有测试
  - 生成代码（`*_gen.go` / proto 产物）

## 3. 错误处理与日志

- **关注点**：错误是否被显式处理、是否有意义的上下文、日志是否合规
- **检查项**：
  - 项目档案摘要中的关键约束（错误处理 / 日志相关条目）是否被遵守
  - `_ = err` / `_, _ = fn()` 显式忽略错误（除非短生命周期 + 注释说明）
  - `panic` / `log.Fatal` 处理常规业务错误
  - 同一错误在多层重复包装 / 重复打印
  - `fmt.Errorf("error: %w", err)` 这种无意义包装
  - 跨服务边界错误码是否按项目约定透传（不二次包装）
  - 日志是否含敏感信息（密码 / token / 完整身份证 / 银行卡号）
  - 业务非阻断错误是否用 ERROR 级别（应为 WARN）
  - 是否每个函数入口都打印日志（噪音）
- **反例**：
  - `if err != nil { panic(err) }` 在常规请求路径
  - `log.Printf("password=%s", req.Password)`
  - 上游能 `cerr.FromGRPCStatus` 解析却包装成自己模块的错误
- **不报的情形**：
  - main 启动失败用 `log.Fatal` 是合理的
  - 测试代码中的 `panic` 在 setup 阶段

## 4. 命名与可读性

- **关注点**：命名是否一致、易读；是否存在 stutter / 过深嵌套 / 魔法数字
- **检查项**：
  - 项目档案摘要中的关键约束（命名相关条目）是否被遵守
  - 类型名是否带冗余层级后缀（如包名是 `service`，类型名 `UserService`）
  - 方法名是否在独立单领域包内 stutter（`config.LoadConfig()`）
  - 单字母变量超出短生命周期循环范围
  - 函数体超过 80-100 行且未拆分
  - 嵌套层级 > 4
  - 魔法数字 / 魔法字符串没有命名常量
  - 同一概念在不同文件用不同词（如 `userId` / `uid` / `user_id` 混用）
- **反例**：
  - `func process(a, b, c int) int { ... }` 单字母参数
  - `if x { if y { if z { if w { ... } } } }` 五层嵌套
  - `time.Sleep(86400 * time.Second)` 没有 `secondsPerDay` 常量
- **不报的情形**：
  - 循环变量 `i, j, k`
  - 短回调里的 `e`（error）/ `r`（result）

## 5. 依赖与第三方库

- **关注点**：依赖是否克制、是否有未使用 / 重复 / 安全风险的依赖
- **检查项**：
  - 项目档案摘要中的关键约束（依赖相关条目）是否被遵守
  - 是否引入了"标准库已能解决"的第三方库（理由是"代码更短""更优雅"）
  - 同一功能是否引入两套库（如 logrus + zap + slog 共存）
  - 已废弃 / 长期未维护的依赖（语言生态可识别的）
  - 大版本明显落后且有已知 CVE 的依赖
  - vendor 目录的修改是否合理
  - 直接 import 内部包的违规（如 Go 的 `internal/` 跨模块 import）
- **反例**：
  - 引入 `github.com/foo/sliceutil` 仅为 `Contains` 一个函数
  - 同时使用 `encoding/json` 和 `json-iterator/go` 没有理由
- **不报的情形**：
  - 大量但都在使用的依赖（除非项目明确要求精简）
  - 仅在测试中使用的 testify 等通用库

## 6. 文档与代码同步

- **关注点**：README / 架构文档 / API 文档 / GoDoc 与代码的事实一致性
- **检查项**：
  - 文档示例 / 命令 / 路径 / 字段名是否与代码一致
  - 文档列出的功能是否在代码中实现
  - 已删除的功能是否仍在文档中描述
  - API 文档中的错误码是否与代码返回一致
  - GoDoc / docstring 是否描述了已不存在的参数
  - 函数 / 类型注释语言是否符合项目要求（CLAUDE.md 通常要求中文）
  - 历史包袱：`迁移指南` 章节但迁移已完成 / `v1 暂时保留` 但 v1 已被删
- **反例**：
  - README 写 `make run-dev` 但 Makefile 没有该目标
  - API 文档写错误码 `1001`，代码返回 `1-003-0001`
  - 文档说"支持 MySQL 与 Postgres"，代码只连 MySQL
- **不报的情形**：
  - 措辞差异（如"用户" vs "客户"）但语义不影响实现
  - 拼写错误（除非批量出现，转给文档审查工具更合适）

## 7. 项目特定约束（动态生成，非固定清单）

主对话把 SKILL.md §2 项目档案摘要中的"关键约束"（项目自身强调的硬约束）转为本维度的检查项，注入 SKILL.md §3.4 Agent 4 的 prompt。这是该 skill 适配所有仓库的关键。

**派工时主对话必须做的转换**：

每条原文约束 → 转为一个具体的"如果命中 X 模式 / 缺失 Y 模式 → 报告"的检查项。例如：

| 原文约束（来源 file:line） | 转化后的检查项 |
|---|---|
| "金额/积分统一 ×100 整数存储和传输，禁止浮点" | 搜索 `float64` / `float32` 出现在金额、价格、积分相关字段（变量名含 `price` / `amount` / `point` / `balance` 等）的位置 |
| "UNIQUE KEY 必须与 deleted 列联合" | 搜索 migrations / schema 中的 `UNIQUE KEY`，检查是否包含 `deleted` 列 |
| "日志级别：业务非阻断错误用 WARN，ERROR 只给系统级阻断故障" | 搜索 `slog.Error` / `log.Error`，检查上下文是否为系统级阻断故障 |
| "禁止使用 npm/yarn/pnpm，统一 bun" | 搜索 `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` 是否存在；搜索 CI 脚本中的 `npm install` / `yarn` |
| "data 层禁止跨库查询/事务" | 搜索 repository 内是否出现多个 db client；搜索是否有跨服务的 `JOIN` |

**派 Agent 4 时**，把转化后的检查项（不要原文罗列）注入 prompt，并明确"每命中一处，给出 file:lines + 证据"。

**优先级**：项目特定约束的违规默认提升一级（项目自己强调的硬约束，违反通常是 🔴 BLOCKER / 🟡 IMPORTANT）。

---

## 派 agent 时的通用 prompt 模板

```text
你是项目质量审视的 <维度名> 专员。仅 read，不修改任何文件、不操作 git。

【审视范围】
<目标目录绝对路径>

【项目档案摘要】（来自 SKILL.md §2，已为你裁剪到与本维度相关的部分）
<嵌入档案摘要中相关的语言栈 / 关键约束 / 项目地图条目>

【本维度检查清单】
<主对话从本文件抽取对应小节并转写为可执行的搜索/检查动作>

【输出格式】（严格遵守，便于汇总，详见 SKILL.md §3.6）
🔴 BLOCKER / 🟡 IMPORTANT / 🟢 NIT
- file:line · 问题简述 → 建议改法（≤2 行）
  违反约束：<约束原文 + 来源 file:line / 或"通用规范">
  影响：<具体落到正确性 / 安全 / 可维护性 / 可读性 哪一项>

【纪律】
- 每条 finding 必须有 file:lines 与原文证据；推断必须显式标注"推断"
- 优先以项目自身约束为准；通用规范作为补充
- 只报真正影响维护、正确性、安全或可读性的问题
- 轻微措辞差异、风格偏好不报
- 不修改任何文件，仅做 review 报告
```
