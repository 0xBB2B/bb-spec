# Spec 需求拆解与文档化

## 目标

把用户脑子里模糊、混在一起的需求，通过对话**澄清 → 拆解 → 文档化**，输出一套互不重叠、各自只讲一件事的"小规则"文档，便于后续逐条实现与验证。

## 核心原则

1. **一文一规则**：每个文档只描述一条独立的功能逻辑或约束，**不超过 100 行**（不含 frontmatter）。
2. **先澄清再写**：在动笔前必须通过对话把假设、边界、并发/异常等约束讲清楚，禁止脑补。
3. **必须举例**：每个文档结尾给出**一个具体的、可被测试的例子**，体现该规则的核心场景。
4. **拒绝大而全**：发现需求中混入多个关注点时，**强制拆解**为多份文档，而不是塞进一个长文档。
5. **中文优先**：文档正文用中文，标识符、API 名、错误码等保持英文。
6. **按需加载结构**：模仿 agent 读取 skill 的机制——每份 spec 文档头部带 `name + description` frontmatter，由 `docs/spec/INDEX.md` 汇成轻量索引；读者（人或 agent）先扫索引判断相关性，再按需打开具体文件。
7. **自主独立**：每份 spec 文档必须自包含，**禁止跨文档相互参考 / 相互引用**。读者只读这一份就能完整理解规则；不允许出现"详见 `xxx.md`"、"复用 A 中定义的 X"、"参见 B 的例子"等表述。同一概念在多处需要时，**各自简短复述一遍**，而不是引到别处去读。INDEX.md 是唯一允许列出其他文档的地方，且只列 name + description，不承载逻辑串联。

## 工作流

### 步骤 0：盘点已有 spec

动手前**必须**先读取索引，再决定是否新增：

```bash
cat docs/spec/INDEX.md 2>/dev/null || ls docs/spec/ 2>/dev/null
```

- **索引/目录不存在或为空**：跳过本步骤，进入步骤 1。
- **存在索引或文档**：
  1. 先读 `INDEX.md`，扫每条目的 `description` 判断与本次需求的相关性。
  2. 对相关条目，再打开对应 `.md` 文件细看 frontmatter 与"目的"段。
  3. 判定覆盖情况：
     - **完全覆盖**：告知用户"已有 `<文件>` 覆盖该需求"，询问是【放弃创建】/【修订原文档】/【新增补充规则】，按用户选择处理。
     - **部分覆盖**：列出已存在的相关文档清单给用户看，再继续步骤 1，并在步骤 2 拆解时**避开**已有规则的范围，只补缺失部分。
     - **完全无关**：正常进入步骤 1。

> ⚠️ 禁止在未读完已有 spec 的情况下直接新建同名/同主题文档，避免规则重复或冲突。
> ⚠️ 没有 `INDEX.md` 但有 `.md` 文件，属于历史遗留：步骤 3 创建新文件时一并补建 INDEX 并把所有旧文档登记进去。

### 步骤 1：理解原始需求

读取用户给出的需求描述。如果用户只说一句话（如"我要做存取款系统"），**不要直接动手**——先在对话里追问以下维度（按需选取，不要全问）：

- **角色与场景**：谁来用？什么时候用？典型流程是什么？
- **数据约束**：涉及哪些字段？哪些必须持久化？金额/时间精度？
- **边界条件**：上下限？空值？超额？余额不足？
- **并发与一致性**：是否多用户/多端同时操作？是否要求强一致？
- **失败处理**：网络中断、重复请求、事务回滚怎么办？
- **依赖**：是否依赖其他系统、外部 API、消息队列？

> ⚠️ 一次只问 2-4 个最关键的问题，避免连环轰炸。

### 步骤 2：判断是否需要拆解

如果用户的需求**同时涉及多个独立逻辑**（典型信号：用"和 / 以及 / 还要 / 同时"连接的多个动词），必须拆解。

举例：
- "做一个存取款系统" → 至少拆成【存款】【取款】【余额查询】三份。
- "用户可以注册、登录、找回密码" → 拆成【注册】【登录】【找回密码】三份。
- "取款必须并发安全且不能超额" → 拆成【取款并发安全】【取款金额校验】两份。

向用户**展示拆解方案**并等待确认（auto mode 下直接给出推荐拆解 + 简短理由）。

### 步骤 3：产出文档

在**当前工作目录**下创建 `docs/spec/` 目录（已存在则复用），按以下命名规则为每条规则生成一份独立 `.md` 文件。可二选一，**同一领域内必须保持一致**，不允许同领域既有扁平又有子目录：

- 扁平：`docs/spec/<领域>-<动作>.md`，例如 `docs/spec/withdraw-concurrency.md`。
- 子目录（相关主题较多时推荐）：`docs/spec/<领域>/<动作>.md`，例如 `docs/spec/withdraw/concurrency.md`、`docs/spec/withdraw/amount-validation.md`。

> 子目录只是物理组织方式，**不改变「自主独立」原则**：同一子目录下的文档之间同样禁止互相引用。
> 子目录内**不放** `INDEX.md` 等额外索引文件，全局索引只有 `docs/spec/INDEX.md` 一份。

每份文档**必须**以 frontmatter 开头（见下方模板），正文遵循固定五块结构。

### 步骤 4：更新索引

每生成或修订一份 spec 文档后，**必须**同步更新 `docs/spec/INDEX.md`：

- 索引文件没有 frontmatter，本身只是个清单。
- 每条目一行，格式：`- [<name>](<相对路径>) — <description>`。
- `name` 与 `description` 直接复制对应文档的 frontmatter，禁止改写。
- 路径为相对 `INDEX.md` 的链接：扁平用 `withdraw-concurrency.md`，子目录用 `withdraw/concurrency.md`。
- 按领域分组（用 `## <领域>` 二级标题），同领域内按文件名字母序；采用子目录时，二级标题与子目录名一致。
- 删除文档时一并删除对应索引行。

### 步骤 5：自检

写完后逐份过一遍：
- [ ] frontmatter 含 `name` 和 `description`？
- [ ] `description` 是一句话（≤ 80 字），能让读者判断要不要读？
- [ ] 正文行数 ≤ 100？
- [ ] 是否只讲一件事？（如果出现两个独立的"必须 / 不能"，再拆）
- [ ] 是否给出了具体例子？
- [ ] 例子是否可被测试 / 可被验证？
- [ ] `INDEX.md` 已同步更新？
- [ ] 文档**未引用**任何其他 spec 文档（无"详见 / 参见 / 复用 xxx.md"），单读这一份即可理解？

## 单文档模板

每份 spec 文档**必须**包含 frontmatter + 正文五块，**不允许新增大段落**：

```markdown
---
name: <kebab-case 与文件名一致，不含 .md>
description: <一句话说明这条规则讲什么，让读者判断要不要读，≤ 80 字>
---

# <规则标题>

## 目的
<一句话说明这条规则要解决什么问题。>

## 逻辑
<3-10 行说明核心逻辑。可以用编号列表或伪代码，但不要贴完整实现。>

## 约束
- <约束 1，例如"必须使用数据库行锁">
- <约束 2，例如"金额单位为分，禁用浮点">

## 例子
<一个具体场景，包含输入、过程、预期结果。>

## 验收
- [ ] <可被测试的验收项 1>
- [ ] <可被测试的验收项 2>
```

```markdown
# Spec 索引

```

## 完整示例：用户说"我要做存取款系统"

**对话澄清后拆出**（节选）：

`docs/spec/withdraw-concurrency.md`

```markdown
---
name: withdraw-concurrency
description: 取款并发安全：行锁 + 锁内重读余额，防多端超额提现
---

# 取款并发安全

## 目的
防止用户在多端同时发起取款请求时，绕过余额校验造成超额提现。

## 逻辑
1. 取款请求进入服务层后，先以 `account_id` 为键获取数据库行锁。
2. 锁内重新读取最新余额，校验 `balance >= amount`。
3. 校验通过后扣减余额并写入流水，事务提交后释放锁。
4. 校验失败立即返回错误码 `A-ACC-1001`（余额不足）。

## 约束
- 必须使用数据库行锁（`SELECT ... FOR UPDATE`），禁用应用层互斥锁。
- 同一账户的取款请求必须串行处理，禁止跨事务读旧余额。
- 锁等待超时 3 秒，超时返回 `A-ACC-1002`（请求繁忙）。

## 例子
账户 A 余额 100 元，用户在手机端和网页端同时发起 80 元取款：
- 请求 1 抢到行锁 → 校验通过 → 余额变 20 → 提交。
- 请求 2 等待锁释放 → 重新读余额 20 → 校验失败 → 返回 `A-ACC-1001`。

## 验收
- [ ] 并发 100 次 80 元取款，余额 100，最终只有 1 次成功。
- [ ] 锁等待超过 3 秒返回 `A-ACC-1002`，且无脏数据写入。
```

## 反面案例（不要这样写）

- ❌ 一份文档同时讲"存款"和"取款"。
- ❌ 文档里塞实现细节（具体 SQL、完整函数代码）。
- ❌ 例子是抽象的"用户进行一次取款"，没有具体数值与预期结果。
- ❌ 没有澄清就直接动手生成文档。
- ❌ 新建文档却不更新 `INDEX.md`，导致索引与文件失同步。
- ❌ `description` 写成"详见正文"或长篇大论——它的职责是让读者**不打开文件**就能判断相关性。
- ❌ 把详细内容塞进 `INDEX.md`——索引只列 name + description，绝不放正文。
- ❌ 跨文档引用：在 A 里写"详见 `B.md`"、"参考 B 的约束"、"复用 B 定义的错误码"。同概念在两份都需要时各自简短复述，而不是引到别处去读。
