# 文档审查技能（review-doc）

审查 Markdown 文档，只抓真正影响读者理解的问题。

**核心原则**：

- **机械可判断的由脚本做到 100% 可信**：满足"直接修复条件"的机械问题由 skill 自动修复，零风险无需逐条确认
- **语义判断完全只读**：4 类语义问题并发派 2 个 agent（Claude + Codex 跨模型）独立审查，只输出报告与处理建议，修不修交给用户
- **跨模型独立**：Claude 与 Codex/GPT-5.5 各跑一份，catch 单模型语义盲点
- **基于事实**：每条发现必须指向 `file:line`
- **抓重点不凑数**：宁可少报，不可滥报

---

## 1. 输入解析与范围确认

`$ARGUMENTS` 为审查范围：

- **单个 `.md` 文件**：直接审查
- **目录**：用 Glob 匹配 `**/*.md` 递归，**必须排除** `node_modules/**`、`.git/**`、`vendor/**`、`dist/**`、`build/**`、`.next/**`、`out/**`
- **未提供**：要求用户指定，不要猜测

**关联文档发现**走两条通道，**只追踪一层**，不递归；外部 URL 不纳入：

- 本地 `.md` 链接：脚本运行后由 `links.referenced` 给出绝对路径列表，直接纳入
- 纯文本"详见 xxx.md"等显式提及：脚本无法识别，由语义层 agent 通读时发现

回显一句话给用户：

```text
审查范围：
- 主文档：<文件清单>
- 关联文档：（脚本跑完后追加）
本轮：跑机械脚本 → 自动修可机械修复的 → 并发派 2 个 agent（含 Codex 跨模型）做语义审查 → 输出报告与处理建议
```

---

## 2. 跑机械脚本

一次性跑完三类机械检查；结果可直接消费，不要再用肉眼扫描这三类问题。脚本路径使用 `$HOME`，确保在任意 CWD 下可用：

```bash
# 输出合并 JSON：{ links, terms, structure }
# - links.broken（reason ∈ file-not-found / anchor-not-found / reference-not-defined）
# - links.external（外部 URL，仅记录不验证）
# - links.referenced（本地 .md 引用绝对路径列表，作为关联文档候选）
# - terms.inconsistentTerms（已跳过代码块/inline/URL，已过滤普通英文词，
#   只保留含全大写/驼峰/数字的术语；每个 variant 含 occurrences 完整位置列表）
# - structure.issues（标题层级跳跃 + 表格列数不匹配）
bun run "$HOME/.claude/skills/review-doc/scripts/run-all.ts" <file1.md> <file2.md> ...
```

把 `links.referenced` 中的关联文档纳入审查范围（只追一层）。

---

## 3. 机械层处理（自动修 + 入待决策队列）

主对话扫描脚本输出，对每条机械问题判断"直接修复条件"。**不逐条问用户**，批量处理。

### 3.1 直接修复条件（三者同时满足）

1. 不涉及修改业务逻辑、设计或功能（仅是断链 / 术语 / 层级 / 表格列数等）
2. 唯一合理解（能想到第二种合理改法就不算唯一）
3. 无明显缺点（不影响其他文档、不改业务语义）

判定细则与各类机械问题的修法见 [references/rules.md](references/rules.md) §一。

### 3.2 满足直接修复条件 → 自动修

直接 Edit 修复，记录到"已修清单"，不打断流程。典型场景：

- 断链 file-not-found 且目标路径可唯一确定（如文件被重命名、明显的拼写错误）
- 断链 anchor-not-found 且 anchor 文本是明显笔误
- 标题层级跳跃，调整到正确层级
- 表格列数不匹配，按上下文明显应补/删
- 术语不一致且**单一胜出拼写明确**（高频或符合项目其它文档惯用法）

### 3.3 不满足直接修复条件 → 入"机械待决策队列"

放入 §5 汇总时的待决策清单，与语义层 agent 发现一起出报告。典型场景：

- 断链有多个候选目标
- 术语两种拼写势均力敌且都合理
- 表格列数差异可能改语义
- 标题层级调整会影响目录结构

### 3.4 自动修复完成后回显

```text
[机械层] 自动修复 N 个：
- [已修] file:line · 类型 · 一句话说明
- [已修] file:line · 类型 · 一句话说明
- ...

机械层待决策 M 个（与语义审查报告一起输出）
```

---

## 4. 语义层并发审查（单消息多 tool call）

**关键纪律**：2 个 agent 必须在**同一条消息内并发发起**，不要串行。

每个 agent 的 prompt 都必须包含：

- 文档清单（主文档 + 关联文档绝对路径）
- 4 类语义问题检查清单（见 [references/rules.md](references/rules.md) §二）
- 输出格式（见 §4.3）
- 一句"只读不改：不要修改任何文件，仅输出 review 报告"
- 一句"只报真正影响读者理解的问题；轻微措辞差异、风格偏好不报"

### 4.1 Agent 1 — Claude 通读语义审查

`subagent_type: general-purpose`

prompt 包含 4 类语义问题的可执行检查动作（参照 [references/rules.md](references/rules.md) §二）：

- **前后矛盾**：同一实体的属性 / 行为 / 流程 / 数值在不同位置描述不一致；前文规则被后文示例违反
- **单一真理来源失败**：同一张表 / 同一个流程 / 同一段配置在多处重复且有差异
- **承诺与实际不符**：目录与实际章节不匹配；声明 N 项实际只写 M 项；开头承诺但正文缺失
- **历史包袱**：对当前读者已无价值的残留（已弃用功能说明、过期过渡期提示、无负责人 TODO、v1/v2 对比章节、已完成的迁移指南）

**判定要点**：必须能指出"位置 A 与位置 B 冲突在 X"；历史包袱类需说明为何对**当前读者**已无价值。

长度上限：≤ 1500 字

### 4.2 Agent 2 — Codex 跨模型独立审查

`subagent_type: codex:codex-rescue`（走 Codex CLI / GPT-5.5）

prompt 包含：

- 同 Agent 1 的范围 / 4 类语义问题清单
- 显式说明："用 codex CLI 启动一次 GPT-5.5 review，目标是找单模型（Claude）易遗漏的文档语义盲点"
- 重点关注：
  1. 跨文档的承诺一致性（A 文档说支持 X，B 文档说不支持）
  2. 隐蔽的历史包袱（Claude 易把"看起来在讲历史"的内容当必要历史保留）
  3. 数值 / 流程 / 命令的细微差异（单模型容易整体扫过去）
- 操作：转述 codex 原始发现，**不要重新编辑或加自己观点**

长度上限：≤ 1200 字

### 4.3 统一输出格式（两个 agent 强制）

```text
🔴 BLOCKER / 🟡 IMPORTANT / 🟢 NIT
- file:line · 类型 · 问题简述
  事实：<位置 A 说 X，位置 B 说 Y / 或具体冲突点，≤ 3 行>
  影响：<读者会误解什么 / 开发者会做错什么决策>
  建议：<修法摘要，≤ 2 行>
```

**严重度判定**：

- 🔴 **BLOCKER**：会直接导致读者错误理解或开发者错误决策（关键流程 / 数值 / 命令冲突）
- 🟡 **IMPORTANT**：明显风险（前后矛盾的非关键描述、单一真理失败、明确的历史包袱）
- 🟢 **NIT**：建议性清理（轻微历史残留、可改可不改的承诺与实际差异）

### 4.4 降级策略

派工前用 `which codex` 探测；失败则只跑 Agent 1，并在最终报告中说明缺失了 Codex 那一份。

---

## 5. 汇总与去重

### 5.1 合并源

把以下三类合并为同一份待决策清单：

- §3.3 机械层待决策队列（无法自动修的机械问题）
- §4.1 Agent 1 发现
- §4.2 Agent 2 发现

### 5.2 去重规则

- 同一 `file:lines` + 相似 title → 合并为一条
- 两个 agent 都点出同一问题 → 合并后**标记 ⭐ 交叉验证**（强信号）
- agent 严重度有分歧时，以**最高**为准，备注分歧
- fuzzy match 阈值偏严，避免误合不同问题

### 5.3 重排

按 🔴 → 🟡 → 🟢，每档内按"是否交叉验证"再排，重新编号。

## 6. 输出报告

详细报告模板见 [references/report-template.md](references/report-template.md)。

### 6.1 概览

```text
文档审查完成

范围：N 个文档（主 X / 关联 Y）
机械层：自动修复 A 个；待决策 B 个
语义层：跑了 K 个 agent（Claude / Codex）
（Codex 不可用时显式说明：本次缺失 Codex 跨模型那一份）

合并去重后共 M 条待决策：
- 🔴 BLOCKER：a 条（其中 ⭐ 交叉验证 a' 条）
- 🟡 IMPORTANT：b 条（其中 ⭐ 交叉验证 b' 条）
- 🟢 NIT：c 条

交叉验证强信号（独立 agent 同时指出）：
- <项 1 简述>
- <项 2 简述>
```

### 6.2 详细报告

按严重度分组，每条按以下格式：

```text
### [🔴/🟡/🟢] <项 N> · <类型> · <one-line 标题>  [⭐ 交叉验证（可选）]

**位置**：`file:lines`
**来源**：[机械待决策] / Agent 1 / Agent 2 / ...
**事实**：<3-5 行说明>
**影响**：<读者误解 / 开发者决策错误，落到具体哪个>
**建议**：<修法摘要，≤ 3 行>
```

### 6.3 处理建议表

```text
建议处理顺序：

| 优先 | 项 | 处理方式 |
| --- | --- | --- |
| 立即 | <项 N>, <项 M> | 主对话单独触发修复 |
| 跟进 | <项 K> | 列 followup issue |
| RA | <项 L> | 在文档前言加 Risk Accept 说明 |

要修哪些？（回复项编号 / "全部 BLOCKER" / "到此结束"）
```

## 7. 跨章节硬约束

- **基于事实**：每条问题必须指向 `file:line`；推断必须显式标注"推断"
- **机械层可自动修**：**仅在满足"直接修复条件"时**由 skill 直接 Edit；不满足条件的机械问题入待决策队列，与语义问题一起出报告
- **语义层完全只读**：本 skill 不对语义问题调 Edit/Write；agent 也不允许；用户决定后续如何修复
- **跨模型独立**：必须包含 codex:codex-rescue 这个 agent；不可用时降级为 1 个 Claude agent 并显式告知
- **prompt 自包含**：agent 看不到本对话，所有上下文必须显式传入
- **并发派工**：语义层 2 个 agent 必须单消息并发，不串行
- **不扩大范围**：只审查指定文档及一层引用
- **跨文件同步**：若机械层自动修复影响代码 / Proto / 配置等其他文件，必须告知；语义问题由用户决定时同样提示
- **方案不引入历史包袱**：建议中禁止"加注释标记 deprecated"、"新旧并列保留"、"暂时保留待后续处理"等过渡式写法
- **语言**：所有输出使用中文

---

## 8. 实现要点（给执行 skill 的 AI 看）

- **§2 → §3 → §4 串行 → 并行**：先跑脚本拿事实，机械层批量处理，再单消息并发派语义 agent
- **prompt 大小控制**：把文档清单 + 4 类语义检查清单内联到 agent prompt，**agent 不读 references/rules.md**（避免重复加载）
- **codex 不可用降级**：Bash 跑 `which codex` 探测；失败则只派 Agent 1
- **交叉验证识别**：merge 时按 `file:lines` + 模糊 title 匹配，阈值偏严
- **机械层批量修**：满足直接修复条件的问题不要逐条问用户，批量 Edit 后一次性回显 `[已修]` 清单
- **不创建分支 / 不 commit / 不 push**：本 skill 不操作 git
- **报告完成态**：最后让用户决定后续动作，本 skill 在此终止；不要主动开始修复

---

## 9. 退出条件

- **正常完成**：输出报告 + 处理建议表 + 等待用户回复
- **用户回复"全部 BLOCKER"或"按推荐执行"**：本 skill 退出，把后续修复交给主对话
- **用户回复"到此结束"**：静默退出
- **agent 失败**：即便 Codex agent 失败，只要 Agent 1 成功也输出可用部分，在报告中说明缺失了哪些 agent
- **机械问题全部自动修完且无语义发现**：明确告知"本轮自动修 N 个机械问题，无语义问题需决策"，本 skill 终止

## 10. 与其他 review skill 的区别

| 项 | review-doc | review-quality | local-ultrareview |
| --- | --- | --- | --- |
| 范围 | .md 文件或目录 | 整仓代码 | base..HEAD 改动 |
| 机械化部分 | 重（脚本输出 100% 可信，自动修） | 无 | 无 |
| 语义 agent 数 | 2（Claude + Codex） | 5（含项目特定专员 + Codex） | 4（含 Codex） |
| 修复 | 机械层自动修，语义层只读 | 完全只读 | 完全只读 |
| 适用 | 文档质量审查 | 项目体检 / 整仓质量审视 | PR / 分支评审 |

文档审查用 review-doc，整仓体检用 review-quality，PR 评审用 local-ultrareview。
