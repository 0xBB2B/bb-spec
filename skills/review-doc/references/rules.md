# 审查规则（rules）

本文件供 SKILL.md 引用：

- §一被 SKILL.md §3 机械层处理使用（决定哪些可自动修、哪些入待决策队列）
- §二被 SKILL.md §4 语义层 agent prompt 使用（注入 4 类语义问题检查清单）
- §三 / §四 / §五是机械层与语义层共享的过滤与边界判定

## 一、哪些算"机械"类（脚本已给出，直接采纳）

修复不改变业务语义，复制脚本给出的事实即可：

- **断链 / 锚点失效 / reference 未定义**（来自 `links.broken`，reason ∈ file-not-found / anchor-not-found / reference-not-defined）
- **英文术语大小写/拼写不一致**（来自 `terms.inconsistentTerms`，已过滤普通英文词，仅保留含全大写/驼峰/数字的术语；每 variant 含 `occurrences` 完整位置列表）
  - 若同时出现在叙述和代码标识符里，**只改叙述部分**，代码标识符保持原样
- **标题层级跳跃**（来自 `structure.issues kind=heading-skip`）
- **表格列数不匹配**（来自 `structure.issues kind=table-column-mismatch`）

机械类问题报告时应给出明确的修复写法（哪个拼写胜出、补哪个标题层级、表格补哪列）。若无法唯一确定（如两种拼写出现次数相当且都合理），**不要自动修**，加入机械待决策队列（SKILL.md §3.3），与语义层报告一起出给用户。

## 二、哪些算"语义"类（需要用户判断）

脚本不可及，必须 LLM 通读：

- **前后矛盾**：同一实体的属性/行为/流程/数值在不同位置描述不一致，或前文定义的规则被后文示例违反
- **单一真理来源失败**：同一张表、同一个流程、同一段配置在多处重复描述且有差异
- **承诺与实际不符**：目录/索引与实际章节不匹配、声明 N 项实际只写 M 项、开头承诺讲的内容在正文缺失
- **历史包袱**：文档保留了对当前读者已无价值的历史内容，应清理。具体形态：
  - 已弃用/已移除功能的残留说明（"旧版支持 X，新版不支持"，且无法切回旧版）
  - 过渡期兼容说明，但过渡早已结束（"暂时保留 X"，暂时是很久以前）
  - "TODO / 待迁移 / 未来会..."之类无负责人无时间表的占位
  - v1/v2 新旧对比章节，仅为历史而非当前决策必须
  - 迁移指南类章节，迁移早已完成，现在只会让新读者困惑

判定要点：必须能指出具体的"位置 A 与位置 B 冲突在 X"，不能凭感觉。历史包袱类需额外说明为何该内容对**当前读者**已无价值。

## 三、过滤（不报）

以下情况不产生报告项：

- **无歧义的措辞差异**：同义词在各自语境中都能自然理解（"创建"和"新建"在不同段落都不会误解）
- **纯排版/格式偏好**：空行数量、列表符号风格、不影响阅读的标题层级小差异
- **理论风险但实际无害**：信息轻微重复且内容完全一致、篇幅很短
- **代码/标识符本身的大小写差异**：变量名 `grpc_client` 与正文 `gRPC` 不构成需要统一的问题

## 四、保留判定（任一即保留）

- 会导致读者产生错误理解
- 会导致开发者做出错误决策
- 引用链接断裂，直接阻碍读者获取信息
- 信息重复且存在实质性差异，维护时容易遗漏

## 五、模糊时的处理

如果不能在 1 句话内说清"正确答案是什么、为什么一定是它"，**不要按机械类自动修**——归入机械待决策队列（与语义问题一起由用户决策），不要假装确定。
