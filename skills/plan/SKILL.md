# Plan 实施计划生成

读取已有的 spec 规格说明 + 项目代码结构，产出一套**自包含、分文件、函数级**的实施计划，让任何 AI 或开发者在没有对话上下文的情况下也能正确编码。

## 与 /spec 的关系

- `/spec` 输出"做什么"：每份 spec 描述一条独立的业务规则与约束。
- `/plan` 输出"怎么做"：每份 plan 描述一个独立的实施单元——需要哪些函数、函数的职责是什么、函数之间如何协作。
- Plan 可以引用 spec 的 `name`（用于溯源），但**必须内联复述**相关的业务规则和约束，禁止"详见 spec"式弱引用。

## 核心原则

1. **一文一单元**：每个 plan 文件只解决一个独立的实施问题（一个模块 / 一个功能 / 一层关注点），不混入不相关的逻辑
2. **自包含可执行**：任何人或 AI 仅凭该文件 + 项目现有代码，无需对话历史或其他 plan 文件，即可正确完成编码
3. **函数级详细**：指明每个需要创建/修改的函数名与职责，但**不写具体参数签名和函数体**——留给执行者根据语言惯例和上下文决定
4. **尊重现有代码**：先摸清项目结构和已有代码，plan 基于真实代码路径和命名风格产出，而非凭空设计
5. **有序但独立**：INDEX.md 给出推荐执行顺序和依赖关系，但每份 plan 文件内部自包含，不依赖"先做完 plan A"才能理解
6. **中文优先**：正文中文，函数名、文件路径、技术术语保持英文
7. **不超过 200 行**：单文件正文（不含 frontmatter）上限 200 行；超过则必须继续拆分

## 输出目录

默认读取 `.bb-channel/docs/spec/`，输出至 `.bb-channel/docs/plan/`。项目根目录 `.bb-channel.yaml` 的 `docs_dir` 可覆盖基础路径（如 `docs_dir: my/docs` → `my/docs/spec/`、`my/docs/plan/`）。

## 工作流

### 步骤 0：前置检查——确认 spec 存在

```bash
cat .bb-channel/docs/spec/INDEX.md 2>/dev/null || echo "NO_SPEC"
```

- **spec 不存在**：告知用户"未找到 `.bb-channel/docs/spec/`，建议先运行 `/spec` 完成需求拆解"，终止流程。
- **spec 存在**：完整读取 `INDEX.md`，然后**逐份打开**所有 spec 文档，理解全部业务规则和约束。禁止只看索引不看正文。

### 步骤 1：摸清项目现状

在写 plan 之前，必须了解项目的真实状态：

1. **技术栈识别**：检查 `go.mod` / `package.json` / `Cargo.toml` / `pyproject.toml` 等，确定语言和框架
2. **目录结构**：用 `find` 或 codegraph 工具了解项目的包/模块组织方式
3. **已有代码**：如果项目不是从零开始，识别已有的关键模块、已实现的功能、可复用的基础设施（数据库连接、HTTP 框架、中间件等）
4. **命名风格**：观察已有代码的命名惯例（驼峰/蛇形、包命名、文件命名），plan 产出的函数名和文件路径必须与之一致

> ⚠️ 如果项目为空（新项目），跳过已有代码分析，但仍需根据 spec 中的技术约束推断合理的项目结构。

### 步骤 2：规划拆分策略

基于 spec 内容和项目结构，决定如何将实施工作拆分为多个独立 plan 文件。拆分维度（按优先级）：

1. **按层拆分**：数据层 / 业务逻辑层 / 接口层（如果每层足够复杂）
2. **按功能拆分**：每个独立业务功能一份 plan（如"存款"和"取款"各一份）
3. **按关注点拆分**：横切关注点单独成文（如"错误处理"、"并发控制"、"数据校验"）

拆分原则：
- 每份 plan 对应**一个可独立完成和验证的实施单元**
- 宁可多拆几份小文件，也不要一份大文件塞多个关注点
- 如果一个 spec 文档的实施本身就很简单（< 5 个函数），可以合并到相关 plan 中，不必一一对应

向用户**展示拆分方案**（列出每份 plan 的文件名和一句话描述），等待确认后再动手写。

### 步骤 3：盘点已有 plan + 冲突分析

```bash
cat .bb-channel/docs/plan/INDEX.md 2>/dev/null || ls .bb-channel/docs/plan/ 2>/dev/null
```

- **不存在**：跳过，进入步骤 4
- **存在**：完整读取所有已有 plan 文档，识别：
  - **冲突**：新 plan 与已有 plan 在同一代码路径上的设计互斥
  - **可复用**：已有 plan 中已设计好的函数/模块可直接复用
  - **需修订**：已有 plan 需要因新 spec 而更新
- 呈现清单给用户，等待裁决

### 步骤 4：产出 plan 文档

在 `.bb-channel/docs/plan/` 下创建文件，命名规则：
- 扁平：`.bb-channel/docs/plan/<主题>-<关注点>.md`
- 子目录：`.bb-channel/docs/plan/<主题>/<关注点>.md`（同主题文件 ≥ 3 时推荐）

每份文档**必须**包含 frontmatter + 正文六块（见下方模板），每块都是自包含的。

**关键要求——函数清单的写法**：
- 每个函数给出：**函数名**、**所在文件路径**、**一句话职责描述**
- 说明函数之间的**调用关系**（谁调用谁）
- 说明函数依赖的**外部接口**（数据库、HTTP 客户端、消息队列等）
- **不写**：具体参数列表、返回值类型、函数体实现
- 函数名和文件路径必须符合项目已有的命名风格

### 步骤 5：更新 INDEX.md

格式与 spec 的 INDEX.md 类似，但增加**执行顺序**和**依赖关系**信息：

- 每条一行：`- [<name>](<路径>) — <description>`
- 用 `## <阶段>` 分组表示推荐执行顺序（阶段 1 先做，阶段 2 后做）
- 同一阶段内的 plan 可并行执行
- 如果某份 plan 依赖另一份的产出，在描述末尾标注 `[依赖: <name>]`

### 步骤 6：自检

写完后逐份检查：
- [ ] 已读完所有 spec 文档（不只是 INDEX.md）？
- [ ] 已摸清项目现有代码结构和命名风格？
- [ ] 每份 plan 只解决一个独立问题？
- [ ] 每份 plan 正文 ≤ 200 行？
- [ ] 函数清单包含：函数名 + 文件路径 + 职责？
- [ ] 调用关系和外部依赖已说明？
- [ ] 没有写具体参数和实现？
- [ ] 一个没有对话上下文的 AI 仅凭此文件能正确编码？
- [ ] 函数名和文件路径符合项目命名风格？
- [ ] INDEX.md 已同步，包含执行顺序和依赖关系？
- [ ] 相关业务规则已内联复述，无"详见 spec"式引用？

## 单文档模板

```markdown
---
name: <kebab-case，与文件名一致>
description: <一句话概括这个实施单元做什么，≤ 80 字>
---

# <实施单元标题>

## 目标
<一句话说明这个实施单元完成后，系统具备什么能力。>

## 业务规则（来源：spec）
<内联复述与本 plan 相关的 spec 规则和约束，3-10 行。标注来源 spec 的 name 以便溯源，但内容必须自包含，读者不需要去翻 spec。>

## 涉及文件
<列出本 plan 会创建或修改的文件路径，每个一行，标注"新建"或"修改"。>

## 函数清单

### <文件路径 1>

| 函数名 | 职责 |
|---|---|
| `FuncA` | 一句话描述 |
| `FuncB` | 一句话描述 |

### <文件路径 2>

| 函数名 | 职责 |
|---|---|
| `FuncC` | 一句话描述 |

## 协作关系
<说明上述函数之间的调用链和数据流向。用简短的文字或列表描述，如：>
- `Handler` 接收请求 → 调用 `Service` 处理业务逻辑
- `Service` 内部调用 `Repository` 读写数据
- `Service` 在写操作前调用 `Validator` 校验

<如有外部依赖（数据库、消息队列、第三方 API），在此说明如何交互。>

## 验证方式
<完成实施后如何验证正确性，2-5 条具体的验证项。>
- [ ] <验证项 1>
- [ ] <验证项 2>
```

## INDEX.md 模板

```markdown
# Plan 索引

> 按阶段分组，同阶段内可并行。读者按阶段顺序执行，每个 plan 文件自包含。

## 阶段 1：基础设施

- [plan-name](file.md) — description

## 阶段 2：核心业务

- [plan-name](file.md) — description [依赖: xxx]

## 阶段 3：接口层

- [plan-name](file.md) — description [依赖: xxx]
```

## 反面案例（不要这样写）

- ❌ 一份 plan 文件同时涉及数据库设计、业务逻辑、HTTP 接口三层
- ❌ 函数清单只写函数名没写职责，或写了详细参数签名和返回类型
- ❌ 写了具体的函数实现代码（哪怕是伪代码也不要超过协作关系中的调用链描述）
- ❌ "详见 `withdraw-concurrency` spec"式引用，未内联复述相关规则
- ❌ "先完成 plan A 才能理解本文档"——每份 plan 必须独立可读
- ❌ 没有摸清项目结构就凭空设计文件路径和函数名
- ❌ spec 不存在就直接开始写 plan
- ❌ 一个 spec 固定对应一个 plan——应按实施关注点拆分，而非按 spec 一一映射
- ❌ INDEX.md 不标注执行顺序和依赖关系
- ❌ 单文件超过 200 行还不拆分
