# Init 反向 Spec 化

`/spec` 是从对话正向产出规则；`/init` 是**从已有代码与文档反向提炼规则**。落点完全对齐 `/spec`：同一份 `INDEX.md`、同一套 frontmatter、同一份 ≤ 100 行的"一文一规则"格式，使得 `/plan`、`/exec`、`/revise` 可直接接力。

## 核心原则

1. **只提炼"规则"，不誊抄"实现"**：写入 spec 的必须是跨实现的硬约束（错误码格式、命名约定、不变量、业务策略、安全 invariant），而不是某个函数怎么写
2. **格式与 /spec 完全对齐**：frontmatter（`name` + `description`）、`${DOCS_DIR}/spec/<领域>/<规则>.md`、`INDEX.md` 按领域分组——任何一处不一致都会破坏后续 `/plan` 闭环
3. **每条规则必须举例**：从代码里挑一处真实场景作为"例子"，附 `file:line` 来源，避免凭空抽象
4. **并发只拆分区，不拆决策**：subagent 只产出**结构化候选规则草案**，主 agent 统一去重、统一落盘、统一与用户对齐——禁止 subagent 直接写文件
5. **用户裁决拆分方案**：分区清单先给用户确认，再派工——避免基于错误切分跑一长串无效 agent
6. **纯净现态**：spec 只写**当前代码已经在执行**的规则，禁写"应该但还没做"或"历史遗留待清理"
7. **中文优先**：正文中文，标识符 / API / 错误码保持英文

## 工作流

### 步骤 0：读取项目配置 + 检测 spec 现状

```bash
cat .bb-spec.yaml 2>/dev/null
```

有 `docs_dir` → 用其值；缺省 → `.bb-spec/docs`。后续记作 `${DOCS_DIR}`，spec 目录即 `${DOCS_DIR}/spec/`。

```bash
ls ${DOCS_DIR}/spec/INDEX.md 2>/dev/null
```

- **不存在 / 为空** → 正常进入步骤 1
- **已存在且非空** → 不要静默覆盖。向用户呈现三选一：
  - `跳过`：直接告知"已有 INDEX.md，未做任何改动"并退出
  - `增量补全`：读完现有 INDEX.md + 全部已有 spec 文件后，仅提炼"既有 spec 未覆盖的规则"，与既有文档**互不重叠**
  - `全量重做`：先 `mv ${DOCS_DIR}/spec ${DOCS_DIR}/spec.bak.<时间戳>` 备份再走全流程；过程中用户可参考 `.bak` 目录

### 步骤 1：项目盘点

主 agent 自己做，目标是产出"分区清单候选"，不下钻到代码细节：

1. 读顶层结构：`ls`、根目录的 `README*`、`CLAUDE.md`、`docs/`、`ARCHITECTURE.md`、`CONTRIBUTING.md`
2. 识别技术栈：`package.json` / `go.mod` / `pyproject.toml` / `Cargo.toml` / `pom.xml` / `Dockerfile` / `*.tf` 等
3. 看模块边界：codegraph 可用就 `codegraph_status` + `codegraph_files <顶层目录>`；否则 `ls -la <主源码目录>`
4. 看 git 主题：`git log --oneline -50` 粗看历史里反复出现的模块名 / 主题词

回显给用户：

```
项目盘点：
- 栈：<识别到的语言/框架/包管理器>
- 主源码目录：<路径>
- README/CLAUDE.md 提示的核心主题：<关键词 1>、<关键词 2>……
- 已有显式文档：<docs/ 下相关文件清单或"无">
```

### 步骤 2：识别"独立功能区"并向用户确认

按以下顺序优先级切分（同一项目可混用）：

1. **架构层**：API / Service / Repository / Domain / Infra 等（Go 三层、DDD 四层都适用）
2. **业务模块**：用户 / 订单 / 支付 / 权限 / 通知……（看顶层目录或 package 命名）
3. **前端特性区**：路由 / 页面 / 组件库 / 状态管理 / API client
4. **横切关注点**：日志 / 错误码 / 配置 / 鉴权 / 限流 / 缓存 / 监控（**单独一区**，跨模块共享的硬约束几乎都落这里）
5. **运维 / CI**：Docker / GitHub Actions / 部署脚本

输出分区候选清单：

```
建议分区（N 个）：
1. <分区名> — <一句话描述> — 大致涉及目录：<路径>
2. ...
横切关注点合并单独一区：<已识别清单>
```

询问用户：`确认 / 调整分区 / 合并 / 拆分？`

### 步骤 3：方案质检（向用户确认前的自检）

逐项问自己，发现问题先调整再展示：

1. **是否有"伪规则"分区**：某分区里能想到的"规则"其实都是某语言/框架自带保证（如"TypeScript 必须有类型注解"）？删
2. **是否漏了横切**：错误码 / 日志格式 / 时间戳精度 / ID 生成 / 命名约定——这些跨模块的硬约束有没有专门一个分区兜？没有就补
3. **是否能合并**：两个分区的预期产出有 >50% 重叠？合
4. **是否一定要分**：项目很小（源码 < 20 文件）就别拆——单个主 agent 一遍提炼到位即可

质检过后展示最终分区给用户，等"确认"再进入步骤 4。

### 步骤 4：并发派工（Explore subagent 提取候选规则）

每个分区派 1 个 `Explore` agent（只读，专搜代码/文档，便宜）。所有派工**同一条消息内**发出，并发执行。

每个 agent 的 prompt 必须自包含，至少包含：

```
任务：扫描分区 "<分区名>" 在以下目录/文件范围内的代码与文档，提炼"当前代码已经在执行"的**规则候选**。

范围：<目录列表 / glob>
项目栈：<语言、框架，便于判断哪些是框架自带保证>
已有 spec 索引（若有，便于避免重复）：
<贴 INDEX.md 内容；无则写"无既有 spec">

输出格式（严格 JSON 或等价 Markdown 表，便于主 agent 合并）：

  [
    {
      "name": "<kebab-case，候选文件名>",
      "domain": "<分区名>",
      "description": "<≤ 80 字>",
      "purpose": "<一句话目的>",
      "logic": "<3-10 行核心逻辑>",
      "constraints": ["<硬约束 1>", "<硬约束 2>"],
      "example": "<具体场景：输入/过程/预期结果>",
      "source_refs": ["<file:line>", "..."],
      "confidence": "high|medium|low",
      "rationale": "<为什么这是规则而不是实现细节>"
    }
  ]

筛选标准（必须全部满足才输出，否则丢弃）：
- 这是跨实现的硬约束，不是某个函数的具体写法
- 不是语言/框架/工具本身就保证的（如 SQL 注入有 ORM 防护、类型安全有编译器保证）
- 在 ≥ 2 处代码出现一致的写法（孤例不抽规则）
- 能用一个真实代码场景作例子

禁止：
- 不要写盘（不要 Write / Edit）
- 不要凭空抽象（每条必须 source_refs 至少 1 个 file:line）
- 不要复述实现细节（如"用 gin.Context 取参"——这是实现不是规则）
```

降级：项目很小或用户在步骤 3 选了"不拆"，主 agent 直接执行一份等价的提取任务，跳过并发。

### 步骤 5：合并、去重、落盘

主 agent 收集所有 subagent 返回的候选清单，做：

1. **跨分区去重**：同名 / 同义的合并为一条，保留 confidence 较高 + source_refs 较多的版本
2. **横切优先**：错误码格式、命名约定等若同时被多个分区识别，统一归入"横切"分区
3. **冲突标注**：若代码本身存在不一致（两处 `file:line` 实现互斥），不要硬选——在候选条目里加 `conflict: <描述>`，进入步骤 7 让用户裁决
4. **格式化为 spec 文档**：每条候选生成一个 `.md` 文件，套用如下模板（与 `/spec` 完全一致）：

```markdown
---
name: <kebab-case，与文件名一致>
description: <≤ 80 字>
---

# <规则标题>

## 目的
<一句话>

## 逻辑
<3-10 行>

## 约束
- <约束 1>
- <约束 2>

## 例子
<具体场景。结尾标注来源：来源：file:line[, file:line]>

## 验收
- [ ] <可测试验收项 1>
- [ ] <可测试验收项 2>
```

路径：`${DOCS_DIR}/spec/<分区>/<name>.md`，**强制按分区建子目录**，与 `/spec` 一致：

- 禁止扁平放置（如 `<分区>-<name>.md` 或裸 `<name>.md`）
- 分区即便只有 1 条规则也建子目录，避免后续条数增长时再迁移
- `<分区>` 用 kebab-case，与 INDEX.md 分组标题一一对应

### 步骤 6：生成 / 更新 INDEX.md

```markdown
# Spec 索引

> 每条一行。读者先扫此页判断相关性，再打开具体文件。

## <分区 1>

- [name](path.md) — description
- ...

## <分区 2>

- ...
```

按分区分组，同分区字母序。若步骤 0 选的是"增量补全"，则在既有 INDEX.md 上**追加**新条目，不动既有条目。

### 步骤 7：跨文档一致性 review + 冲突清单

落盘后，主 agent 自己快速扫一遍：

- 同一术语在多文档命名是否一致（用户 vs User vs Account）
- 约束之间是否互斥（A 文档说"超时 5s"，B 文档说"超时 10s"）
- 例子是否冲突

把发现的不一致 + 步骤 5 标注的代码冲突合并成清单输出，让用户裁决——**禁止自作主张改任何一份 spec**。

### 步骤 8：完成简报

```
## /init 完成简报

- 模式：全新生成 / 增量补全 / 全量重做（已备份至 <bak 路径>）
- 分区：N 个（<逐项列出>）
- 派工：M 个 Explore agent（并发） / 单 agent 直跑
- 产出：新增 K 条 spec
  - <分区>/<name> — <description>
  - ...
- 跳过的候选：J 条（理由：框架自带保证 / 孤例 / 实现细节）
- 待用户裁决冲突：L 条
  - <一句话描述> — 涉及 <file:line> vs <file:line>
- 下一步：
  - 解决冲突后，可运行 `/plan` 基于这套 spec 生成具体实施计划
  - 或运行 `/spec` 补充本次未覆盖的新需求
```

---

## 硬约束

1. **绝不静默覆盖既有 INDEX.md**：步骤 0 必须三选一并由用户拍板
2. **subagent 只产候选、不写盘**：所有 Write/Edit 由主 agent 串行执行，保证格式一致
3. **每条 spec 必须有 `source_refs`**：例子段落末尾标注 `来源：file:line`，无来源不准落盘
4. **单文件 ≤ 100 行**：超过即拆，与 `/spec` 一致
5. **禁止跨文档引用**：每份 spec 自包含，不写"详见 X.md"——与 `/spec` 一致
6. **不写"应然"只写"实然"**：代码没在做的事不进 spec；若用户希望补"该有但还没有"的规则，提示走 `/spec` 正向流程
7. **冲突不自决**：步骤 5/7 的冲突全部抛给用户
8. **强制子目录结构**：所有 spec 必须落在 `${DOCS_DIR}/spec/<分区>/<name>.md`，禁止扁平根目录文件——与 `/spec` 一致

---

## 与 /spec 的边界

- `/init` 只在"项目首次接入 bb-spec 工作流"或"用户主动要求批量反向梳理"时使用
- 单条新增 / 修订规则用 `/spec`，不要拿 `/init` 当日常工具
- `/init` 跑完后，后续维护一律走 `/spec`（编辑既有文件 + 跨文档 review）
